# ТЗ клиенту (Unity): пуш-уведомления (FCM)

Контекст: запускаем **пользовательские пуш-уведомления** (ретеншн, анонсы сказок, реактивация).
Управляются из админ-панели на стороне бэкенда. Канал — **FCM** (тот же Firebase-проект, что и
аналитика). Backend-часть описана в `DEV_PLAN_PUSH_NOTIFICATIONS.md`.

**Без этой доработки пуши физически не будут доходить до пользователей** — регистрация токена и
обработка входящих возможны только в приложении.

---

## Что нужно сделать

### 1. Подключить Firebase Cloud Messaging
- Добавить **Firebase Messaging** в Unity SDK (Firebase уже в проекте ради аналитики — добавляется
  модуль Messaging).
- **iOS:** включить capability **Push Notifications** и **Background Modes → Remote notifications**;
  APNs-ключ (.p8) заливаем мы в Firebase Console (со стороны клиента ничего, кроме capabilities).
- **Android:** FCM работает «из коробки»; создать **notification channel**; на **Android 13+**
  запросить рантайм-разрешение `POST_NOTIFICATIONS`.

### 2. Запросить разрешение на уведомления
- iOS — явный системный запрос (`UNUserNotificationCenter` / Firebase wrapper). Показать в удачный
  момент (после онбординга, не на первом же экране).
- Android 13+ — `POST_NOTIFICATIONS`.
- Это **не** ATT/реклама — детский комплаенс не нарушается, IDFA не запрашиваем.

### 3. Зарегистрировать токен на нашем бэкенде
Получить FCM registration token и отправить:

```
POST {API_URL}/api/push/register
Content-Type: application/json

{
  "userId":     "<тот же device-based userId, что и везде>",
  "token":      "<FCM registration token>",
  "platform":   "ios" | "android",
  "appVersion": "<версия приложения>",
  "lang":       "ru" | "kz" | "en"
}
```

Когда вызывать:
- при старте приложения (если разрешение дано);
- **на каждый refresh токена** (FCM периодически его обновляет — обязательно перерегистрировать);
- при смене языка в приложении — повторный `register` с новым `lang` (чтобы пуши приходили на
  правильном языке).

При выключении пушей пользователем или разлогине:
```
POST {API_URL}/api/push/unregister
{ "token": "<текущий токен>" }
```

### 4. Обработать входящий пуш и deep-link
В пуше в поле **`data`** приходит маршрут. Контракт:

```json
{
  "type":   "tale" | "paywall" | "url" | "home",
  "taleId": "<id сказки>",      // если type=tale
  "url":    "<внешняя ссылка>"  // если type=url
}
```

Поведение по тапу:
| `type` | Действие |
|--------|----------|
| `tale` | открыть экран сказки `taleId` |
| `paywall` | открыть экран подписки/пейволл |
| `url` | открыть ссылку (внутренний webview или внешний браузер) |
| `home` | просто открыть приложение (главный экран) |

Обработать оба случая: приложение **в фоне/закрыто** (тап по уведомлению) и **на переднем плане**
(показать/не показывать баннер — на ваше усмотрение, но deep-link по тапу обязателен).

### 5. (Опционально) Репорт открытий для статистики
Если несложно — при открытии по пушу дёрнуть:
```
POST {API_URL}/api/push/opened
{ "campaignId": "<из data.campaignId>", "userId": "<userId>" }
```
(В `data` тогда добавим `campaignId`.) Это даст метрику open-rate в панели. Не критично для запуска.

## Чего делать НЕ надо
- Никакого своего UI управления кампаниями — всё на бэкенде/сайте.
- Не хранить и не показывать PII в пушах.
- Не запрашивать IDFA/рекламные идентификаторы ради пушей.

---

## Что нужно от нас (backend) вам для интеграции
- `API_URL` (боевой и тестовый) — есть.
- Подтверждение, что APNs-ключ залит в Firebase (сделаем в Фазе 0).
- Согласование финального формата `data`/deep-link (выше — предложение, дошлифуем вместе).

## Чек-лист готовности клиента
- [ ] Разрешение запрашивается, токен получен.
- [ ] Токен уходит в `/api/push/register` при старте, refresh и смене языка.
- [ ] Тестовый пуш с бэкенда приходит на iOS и Android.
- [ ] Тап открывает нужный экран для всех `type` (tale/paywall/url/home).
- [ ] Пуш приходит на языке пользователя.
- [ ] Выключение пушей → `/api/push/unregister`.
