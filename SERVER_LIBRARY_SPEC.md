# Backend: Library (Tale Catalog) — управление каталогом сказок

Цель: **полный серверный контроль над библиотекой** — добавлять новые сказки, помечать
«скоро», прятать и удалять (с очисткой кэша на клиенте). Каталог уже серверный
(`GET /api/tales`), здесь описано, что достроить.

Продуктовые решения (согласованы):
- **Загрузка сказок — по тапу** (как сейчас): клиент показывает карточку сразу, а контент
  качает при открытии. Фоновой автозагрузки НЕ делаем.
- **«Скоро» — ручной флаг** `comingSoon` (без даты/таймеров). Переключаете на сервере.
- **Удаление — «скрыть + удалить кэш»**: сервер помечает сказку `status:"removed"`,
  клиент прячет её И стирает локальные файлы.

Базовый URL: `https://bala-stories.apiapp.kz:3000`

---

## 1. Модель данных

Таблица `tales` (каталог). Поля должны 1:1 ложиться в клиентский `TaleSummary`:

```sql
CREATE TABLE tales (
  id             TEXT PRIMARY KEY,       -- напр. "golden_egg"
  title          TEXT NOT NULL,          -- дефолтный заголовок (fallback)
  titles         JSONB,                  -- { "ru": "...", "kz": "...", "en": "...", "uz": "..." }
  free           BOOLEAN NOT NULL DEFAULT FALSE,
  coming_soon    BOOLEAN NOT NULL DEFAULT FALSE,
  status         TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'hidden' | 'removed'
  has_default_narration BOOLEAN NOT NULL DEFAULT FALSE,
  cover_url      TEXT,
  download_size  BIGINT NOT NULL DEFAULT 0,       -- байты, для прогресс-бара
  sort_order     INT NOT NULL DEFAULT 0,          -- порядок в библиотеке (меньше = выше)
  content_version INT NOT NULL DEFAULT 1,         -- ↑ при изменении контента (см. §5)
  langs          TEXT[] ,                          -- в каких языках сказка реально есть
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Значения `status`:
- `active` — показывается нормально.
- `hidden` — не показывается новым, но локальный кэш у скачавших НЕ трогается (грейс).
- `removed` — не показывается И клиент удаляет локальный кэш этой сказки.

## 2. GET /api/tales?lang=xx — контракт (расширяем существующий)

Уже используется клиентом. Возвращает массив. **Добавить поле `status`** (и убедиться, что
`comingSoon` присутствует). Пример элемента:

```json
{
  "id": "golden_egg",
  "title": "Золотое яйцо",
  "titles": { "ru": "Золотое яйцо", "kz": "Алтын жұмыртқа", "en": "The Golden Egg", "uz": "..." },
  "lang": "ru",
  "file": null,
  "hasDefaultNarration": true,
  "free": true,
  "coverUrl": "https://.../covers/golden_egg.jpg",
  "bundled": false,
  "downloadSize": 5242880,
  "comingSoon": false,
  "status": "active"            // NEW: active | hidden | removed
}
```

Правила выдачи:
- Возвращать сказки в порядке `sort_order` (клиент дополнительно сортирует по доступности).
- **`status:"removed"` НАДО отдавать ещё какое-то время** (например 30 дней после удаления) —
  иначе клиент не узнает, что сказку надо стереть (он реагирует на явный `removed`, а не на
  «пропала из списка»). После окна ретеншена запись можно перестать отдавать/удалить из БД.
- `hidden` — можно не отдавать вовсе (тогда просто не показывается) ИЛИ отдавать со
  `status:"hidden"` (клиент сам скроет). Оба варианта клиент поддерживает.
- Не фильтровать `comingSoon` на сервере — клиент сам покажет бейдж «скоро».

> ВАЖНО про язык: клиент при УСПЕШНОМ ответе сервера считает список авторитетным и может
> удалить кэш `removed`-сказок. Поэтому `GET /api/tales?lang=` должен возвращать ВСЕ сказки
> каталога (с локализованными заголовками через `titles`), а не только переведённые на `lang`.
> Иначе сказка, отсутствующая в одном языке, не должна получать `status:"removed"` — статус
> `removed` шлём только для реально удалённых.

## 3. Админ-эндпоинты (защита — админ-ключ/роль)

| Метод | Эндпоинт | Тело | Действие |
|-------|----------|------|----------|
| GET    | `/api/admin/tales` | — | Полный список включая hidden/removed, для управления. |
| POST   | `/api/admin/tales` | объект сказки | Создать сказку (загрузка контента — см. §4). |
| PATCH  | `/api/admin/tales/{id}` | частичные поля | Обновить: `free`, `comingSoon`, `status`, `titles`, `sortOrder`, `coverUrl`… |
| POST   | `/api/admin/tales/{id}/coming-soon` | `{ "value": true }` | Быстрый тумблер «скоро». |
| POST   | `/api/admin/tales/{id}/publish` | — | `comingSoon=false, status='active'`. |
| DELETE | `/api/admin/tales/{id}` | — | `status='removed'`, `updated_at=now()` (мягкое удаление; клиент подчистит кэш). |
| POST   | `/api/admin/tales/{id}/reorder` | `{ "sortOrder": 5 }` | Порядок в библиотеке. |

Удаление — **мягкое** (`status='removed'`), НЕ физическое стирание строки сразу: строка
нужна в выдаче на время окна ретеншена (§2), чтобы клиенты успели подчистить кэш.

## 4. Контент сказки (страницы, иллюстрации, обложка)

Клиент качает контент по существующим эндпоинтам (не меняются):
- `GET /api/tales/{id}?lang=` — текст/страницы (`TaleDetail`: `pages`, `genderedPages`, `totalPages`).
- `GET /api/tales/{id}/illustration/{page}?gender=` — иллюстрация страницы (с фолбэком по гендеру).
- обложка — по `coverUrl`.

Чтобы «добавить новую сказку» = завести строку в `tales` (§1) + залить её текст и
иллюстрации в то же хранилище/формат, что у существующих (напр. `golden_egg`). Как только
запись появилась в `GET /api/tales`, клиент показывает карточку и качает по тапу.

Требования к контенту (из опыта, см. заметки проекта):
- Для gendered-страниц заливать ОБА варианта `page_N_boy` и `page_N_girl`.
- Страница 0 обязана существовать хотя бы в одном варианте (plain / _boy / _girl), иначе
  клиент сочтёт сказку «не скачанной» и будет пытаться качать снова.
- `downloadSize` заполнять реальным размером (для прогресс-бара).

## 5. (Опционально) Версионирование контента

Если нужно ОБНОВИТЬ уже выпущенную сказку (перерисовали страницу, поправили текст) и чтобы
клиенты перекачали — увеличьте `content_version` в записи. Клиент сравнивает с версией в кэше и
при расхождении до-качивает/обновляет. Для MVP можно не делать; заложено на будущее.
(Если внедряем — добавить `contentVersion` в `GET /api/tales` и в `.ready` метку на клиенте.)

## 6. Что делает КЛИЕНТ (уже реализовано)

- На старте `LibraryScreen` дергает `GET /api/tales?lang=`.
- При успешном ответе сервер = источник правды:
  - для каждой сказки `status:"removed"` → `AssetCache.DeleteTale(id)` (стирает
    illustrations/text/cover/narration/ai-narration/ready);
  - из грида убираются `removed` и `hidden`;
  - `comingSoon` показывается бейджем, карточка некликабельна.
- При НЕуспехе (офлайн) — фолбэк на bundled-манифест, кэш НЕ трогается (без ложных чисток).
- Новые сказки появляются автоматически (просто новая запись в каталоге), качаются по тапу.

Клиентские точки (для справки): `AssetCache.DeleteTale`, `TaleSummary.status/IsRemoved/IsHidden`,
`LibraryScreen.ReconcileLibrary` / `VisibleTales`.

## 7. Чек-лист приёмки

- [ ] Новая запись в `tales` → появляется в `GET /api/tales` → карточка в приложении, качается по тапу.
- [ ] `comingSoon=true` → бейдж «скоро», карточка некликабельна.
- [ ] `status='removed'` → у клиента сказка исчезает И локальный кэш стёрт (проверить, что
      после удаления освободилось место / файлы пропали).
- [ ] `status='hidden'` → сказка исчезает, но уже скачавший может открыть (если так решим).
- [ ] Офлайн (сервер недоступен) → библиотека из bundled-манифеста, ничего не удаляется.
- [ ] `removed`-запись отдаётся ещё N дней (ретеншн), потом убирается из выдачи.
- [ ] `sort_order` управляет порядком карточек.
