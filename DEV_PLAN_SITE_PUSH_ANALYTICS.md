# План разработки: Админ-сайт (библиотека + подписки + пуш-алерты) + Firebase-аналитика

Статус: план (код не писан). Идём в прод — цель «железно», без костылей, с подробными
дебаг-логами на каждом шаге.

## Согласованные решения (вводные от продукта)

| Вопрос | Решение |
|--------|---------|
| Пуш-уведомления | **Только алерты админам** (покупка/возврат/ключевые события). Пользовательских кампаний нет → **клиент не трогаем** |
| Firebase-проект | **Существует** (доступ будет предоставлен) — используем для аналитики |
| Где админка | **В сайте `bala-stories`** (существующая промо-админка, TS/Express/Prisma). Fairy-бэкенд остаётся источником правды и хранилищем контента |
| Порядок | **Библиотека → пуши → аналитика** |

## Архитектура (кто с кем говорит)

```
Unity-клиент ──(контракт не меняется)──▶ Fairy-бэкенд (:3000, Node/Express/pg)
                                             │   источник правды: entitlements, tales,
                                             │   файлы контента (data/…), S2S, promo
                                             ▼
Браузер (админ) ─▶ bala-stories сайт ─▶ bala-stories server (TS/Express/Prisma)
   (логин админа)     (frontend)          = BFF/прокси:
                                          - аутентифицирует человека (свой admin-JWT)
                                          - проксирует в Fairy admin API с X-Admin-Key
                                            (ключ живёт ТОЛЬКО на сервере, не в браузере)
                                             │
                                             ▼
                                          Fairy-бэкенд /api/admin/*  ─▶ Postgres + FS

Fairy-бэкенд ──(алерты)──▶ Telegram-бот (и лента «Активность» в админке)
Unity-клиент ──(события)──▶ Firebase GA4 ──▶ BigQuery (аналитика)
```

Ключевой принцип: **X-Admin-Key никогда не попадает в браузер.** Человек логинится в
bala-сайте (уже есть логин/роли), а bala-server дергает Fairy admin API серверным ключом.

## Фаза 1 — Полное управление библиотекой через сайт

Цель: завести/отредактировать/удалить сказку целиком из браузера (текст + обложка +
иллюстрации обоих гендеров), **не трогая клиент**. Клиент уже реагирует на
`GET /api/tales` (новые появляются сами, `comingSoon`/`status`/`downloadSize` читаются) —
проверено ранее. Значит вся работа — сервер + сайт.

### 1A. Fairy-бэкенд: догрузить недостающее

**Загрузка изображений** (используем уже установленные `multer` + `sharp`; конвертация в
WebP как в `middleware/webpConverter.js`). Все — под `X-Admin-Key`, кладут в существующую
структуру `data/covers/{id}.*`, `data/illustrations/{id}/page_N[_boy|_girl].*`:

| Метод | Эндпоинт | Действие |
|-------|----------|----------|
| POST | `/api/admin/tales/{id}/cover` (multipart `file`) | Залить/заменить обложку → WebP |
| POST | `/api/admin/tales/{id}/illustration/{page}?gender=boy\|girl` (multipart `file`) | Залить страницу (или гендер-вариант) → WebP |
| DELETE | `/api/admin/tales/{id}/illustration/{page}?gender=` | Удалить иллюстрацию |
| POST | `/api/admin/tales/{id}/illustrations` (multipart, zip) | Пакетная заливка (опционально, ускоряет) |
| PUT | `/api/admin/tales/{id}/pages?lang=` (`{ "pages": [...] }`) | Задать/заменить текст страниц языка |

**Валидатор контента** — ловит грабли из LibrarySpec §4 (иначе клиент решит «сказка не
скачана» и будет качать по кругу):

| Метод | Эндпоинт | Что проверяет |
|-------|----------|---------------|
| GET | `/api/admin/tales/{id}/content-check` | есть ли `page_0` (хоть в одном варианте); для gendered-страниц залиты ОБА `_boy` и `_girl`; есть обложка; `downloadSize` пересчитан; какие языки заполнены; расхождение числа страниц по языкам |

Ответ — структурированный отчёт `{ ok, issues: [...], warnings: [...] }`, который сайт
показывает светофором до «публикации».

**Консолидация «coming soon».** Сейчас два источника: `data/coming-soon.json` (файл) и
`tales.coming_soon` (БД, добавлено миграцией 006). Для единого управления с сайта —
перенести записи из файла в БД (одноразовый скрипт), дальше «скоро» = только флаг в БД.
Файл оставить как read-only фолбэк на переходный период. **Клиент не трогаем** — он читает
`comingSoon` из ответа списка независимо от источника.

**Пересчёт `downloadSize`.** После любой заливки иллюстраций — инвалидация кэша размера
(сейчас `downloadSizeCache` в `talesService.js`) и запись реального размера в `tales`.

**Дебаг-логи:** тег `[ADMIN]` на каждую мутацию — что за сказка, какой файл, размер до/после
WebP, путь, кто (через прокси прилетит идентификатор админа в заголовке).

### 1B. bala-stories server: BFF/прокси

- Новый роутер (напр. `src/routes/catalog.ts`) под существующим `authenticateToken +
  requireRole('admin')`.
- Каждый метод проксирует в Fairy admin API, подставляя `X-Admin-Key` из env. Для
  multipart — потоковый passthrough (busboy/streamed multer), чтобы картинки не буферить
  в память целиком.
- Пробрасывать идентификатор админа (`req.user`) в Fairy заголовком `X-Admin-Actor` для
  аудита «кто залил».
- Дебаг-логи `[PROXY]`: метод, путь, статус ответа Fairy, длительность, `requestId`.

### 1C. bala-stories frontend: вкладка «Сказки»

Переиспользуем существующий паттерн (хелпер `api()`, модалки, таблицы, `style.css`):

- **Список** сказок (из `/api/admin/tales`): id, титулы, статус (active/hidden/removed),
  comingSoon, sortOrder, языки, флаг «контент ок» из content-check.
- **Создание/редактирование** (модалка/страница):
  - id, `titles` по языкам, `free`, `sortOrder`;
  - редактор текста постранично на каждый язык (`PUT …/pages`);
  - загрузка обложки (preview);
  - загрузка иллюстраций постранично, отдельно `boy`/`girl`, с превью и индикатором
    «страница 0 есть», «оба гендера есть»;
  - тумблеры: `comingSoon`, `publish`, `hidden`, `removed` (мягкое удаление);
  - reorder (перетаскивание или числом);
  - кнопка «Проверить контент» → показывает отчёт content-check перед публикацией.

### 1D. Чек-лист приёмки (библиотека)

- [ ] Полностью создать сказку из браузера (текст все языки + обложка + иллюстрации
      обоих гендеров + page_0) → появляется в приложении, качается по тапу.
- [ ] `content-check` ловит отсутствие page_0 и непарные гендер-страницы.
- [ ] `comingSoon` → бейдж «Скоро» в приложении, карточка некликабельна.
- [ ] `removed` → в приложении исчезает и локальный кэш стирается.
- [ ] `reorder` → порядок в библиотеке меняется.
- [ ] Офлайн-клиент не ломается (bundled-фолбэк) — не регрессировали.
- [ ] Все мутации видны в логах `[ADMIN]`/`[PROXY]`.

**Изменения клиента: НЕТ.** (Проверить лишь, что в клиенте нет захардкоженного списка
сказок — если есть, это отдельный тикет клиенту, но по LibrarySpec список серверный.)

## Фаза 2 — Пуш-алерты админам

Цель: как только кто-то купил подписку (или возврат/отмена) — мы сразу видим. Без
пользовательских пушей, без изменений клиента.

### 2A. Точки-триггеры (Fairy-бэкенд)

Алерт шлём **только на переход состояния**, не на каждый `/status` (иначе спам):

| Событие | Где ловим | Текст алерта |
|---------|-----------|--------------|
| Новая покупка | `validate` успех, когда было `none/expired` → стало `active` | «💰 Новая подписка: {product} {env} user={id}» |
| Продление | S2S `DID_RENEW` | «🔁 Продление {product}» (можно приглушить/агрегировать) |
| Возврат/отмена | S2S `REFUND`/`REVOKE`/`EXPIRED` | «⚠️ Возврат/отмена {product} user={id}» |
| Промо/admin-грант | `promo` успех / admin grant | «🎁 Промо/грант user={id}» (опц.) |

Идемпотентность: дедуп по `originalTransactionId + notificationType` (не слать дубли при
повторных S2S/ретраях).

### 2B. Доставка

- **Основной канал — Telegram-бот** (env `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`).
  Мгновенно, надёжно, ноль клиента. Модуль `services/alerts.js`, вызывается из точек 2A.
- **Лента в админке** — сохраняем алерты в таблицу (или переиспользуем
  `subscription_events`) + `GET /api/admin/alerts?since=` для страницы «Активность» с
  бейджем непрочитанного.
- Опционально позже: **FCM web-push** в браузер админа (Firebase-проект уже есть) — если
  Telegram будет мало.

### 2C. Сайт: страница «Активность/Продажи»

Лента последних покупок/возвратов (поллинг раз в N сек), бейдж непрочитанного,
фильтр по типу события. Данные — через bala BFF из `/api/admin/alerts`.

### 2D. Дебаг-логи

`[PUSH]` на каждую попытку: тип события, канал, HTTP-ответ Telegram, успех/ошибка, дедуп-хит.

### 2E. Чек-лист приёмки (пуши)

- [ ] Тестовая покупка (sandbox) → Telegram-пинг + запись в ленте за секунды.
- [ ] Возврат (S2S) → алерт.
- [ ] Повторный S2S того же события → **без** дубля алерта.
- [ ] Сбой Telegram не роняет основной флоу (fire-and-forget + лог ошибки).

**Изменения клиента: НЕТ.**

## Фаза 3 — Firebase-аналитика

Аналитика по своей природе **клиентская** (событие про «докуда дочитал» знает только
приложение). Здесь backend вторичен.

### 3A. Работа на КЛИЕНТЕ (Unity) — ТРЕБУЕТСЯ, отдельный тикет клиенту

- Подключить Firebase Unity SDK (GA4), **детский режим**: отключить сбор Advertising ID /
  персонализацию рекламы (требование Google для детских приложений).
- Инструментировать события по таксономии (см. отдельный `ANALYTICS_SPEC.md`, который
  подготовлю): чтение (`tale_open`, `tale_page_view` с `page_index`/`percent`,
  `tale_complete`, `tale_abandon`), библиотека, персонализация/озвучка, монетизация,
  онбординг, ошибки. Плюс user properties (`is_premium`, `premium_source`, `app_language`,
  `has_cloned_voice`).
- **НЕ отправлять PII** (имя ребёнка) — только `child_gender` как параметр.

### 3B. Настройка проекта (Firebase — есть)

- Добавить в проект iOS-приложение `com.mozz.fairyTales` и Android `com.tokengc.balastories`
  (`GoogleService-Info.plist` / `google-services.json` в клиент).
- Включить **BigQuery-экспорт** сразу — развяжет руки для SQL-аналитики.
- QA событий через Firebase **DebugView** до релиза.

### 3C. Backend (опционально — своя копия данных)

- Клиент может дублировать критичные события в наш `POST /api/debug/log` или новый
  `POST /api/analytics/event` (структурированно) → своя аналитика независимо от Google.
  Не блокирует Фазу 3, делаем если нужно.

### 3D. Приватность/комплаенс (критично — детское приложение)

- Отключить ad personalization; consent-флоу; соответствие COPPA / GDPR-K. Согласовать
  до релиза, иначе риск блокировки в сторах.

### 3E. Дашборды (минимум)

- Воронка `tale_page_view` по `page_index` (глубина дочитывания — главный запрос).
- Воронка `paywall_view → purchase_success` (деньги).
- Retention по `is_premium`.

### 3F. Чек-лист приёмки (аналитика)

- [ ] События видны в DebugView; параметры корректны.
- [ ] BigQuery-экспорт наполняется.
- [ ] Собраны 3 базовых отчёта.
- [ ] Детский режим включён, PII не улетает.

**Изменения клиента: ТРЕБУЮТСЯ** (SDK + инструментирование) — это ядро фазы.

## Сквозное: стандарт дебаг-логов

- Теги: `[IAP]` (покупки/S2S), `[ADMIN]` (мутации каталога/подписок), `[PROXY]` (bala BFF),
  `[PUSH]` (алерты), `[HTTP]`/`[HTTP!]` (запросы), `[BOOT]`/`[FATAL]` (старт/краши).
- Сквозной `requestId` (генерит bala BFF, прокидывает в Fairy заголовком) — чтобы связать
  «клик в админке → прокси → операция в Fairy → результат» одной цепочкой.
- Секреты (чеки/токены/ключи/подписи) — маскировать `<redacted len=N>` (уже есть в
  `middleware/requestLog.js`).
- `GET /api/admin/debug/overview` расширить: последние операции каталога и последние алерты.
- Тумблер `DEBUG_HTTP=0` для приглушения после стабилизации.

## Риски / открытые вопросы

1. **Кросс-бэкенд загрузка картинок** через bala-прокси: следить за лимитами размера и
   таймаутами, стримить multipart (не буферить). Альтернатива — прямая заливка в Fairy с
   временным upload-токеном, если прокси окажется узким местом.
2. **Миграция `coming-soon.json` → БД**: одноразовый скрипт + проверка, что клиент не
   регрессировал по «скоро».
3. **Firebase детский комплаенс**: нужен сознательный sign-off до прод-релиза аналитики.
4. **Telegram как единственный канал алертов**: если нужен SLA — добавить второй канал
   (email/FCM). Пока достаточно Telegram + лента.

## Порядок работ (по приоритету)

1. **Фаза 1 — Библиотека** (Fairy upload+content-check → bala BFF → вкладка «Сказки»).
2. **Фаза 2 — Пуш-алерты** (триггеры → Telegram → лента «Активность»).
3. **Фаза 3 — Аналитика** (Firebase setup → `ANALYTICS_SPEC.md` для клиента → дашборды).

Каждая фаза — со своим чек-листом приёмки выше; в прод катим пофазно.
